# Medical Transcript Processing Pipeline – Complete Summary

## 1. Project Overview

The **Medical Transcript Processing Pipeline** is a modular post-processing system for medical ASR transcripts.

- It enhances transcript quality after speech-to-text conversion.
- The pipeline accepts transcript JSON as input.
- Multiple NLP processing stages are executed sequentially.
- Final outputs include:
  - Corrected transcript
  - Structured clinical summary

The project follows a modular architecture where each component has a single responsibility. It is designed specifically for healthcare AI applications.

---

## 2. Objectives

- Improve transcription accuracy
- Preserve timestamps and speaker labels
- Correct fragmented medical terminology
- Support multilingual conversations
- Align outputs from multiple ASR systems
- Generate clinician-ready transcripts
- Produce structured clinical summaries
- Maintain a modular and extensible architecture

## 6. Step 0.5 – Translator

**Purpose**

- Detect non-English text
- Translate only genuine foreign-language phrases
- Preserve medical terminology

**Libraries Used**

- Lingua
- wordfreq
- deep-translator

**Workflow**

```
Read Sentence
  ↓
Detect Language
  ↓
Confidence Check
  ↓
Medical Word Validation
  ↓
Translate if Required
  ↓
Return English Text
```

**Advantages**

- Supports multilingual consultations
- Prevents translation of medical terms
- Preserves abbreviations and drug names
- Produces cleaner transcripts

**Complexity**

| Metric | Value |
|---|---|
| Time Complexity | O(n) |
| Memory Complexity | O(1) |

## 7. Step 1 – Light Normalizer

**Purpose**

Standardizes transcript formatting while preserving metadata.

**Operations**

- Convert to lowercase
- Remove unnecessary punctuation
- Trim whitespace
- Preserve timestamps
- Preserve speaker labels
- Preserve confidence scores

**Workflow**

```
Read Token
  ↓
Lowercase
  ↓
Clean Text
  ↓
Trim Spaces
  ↓
Return Normalized Token
```

**Advantages**

- Improves alignment accuracy
- Produces consistent transcript formatting
- Preserves medical symbols

**Complexity**

| Metric | Value |
|---|---|
| Time Complexity | O(n) |
| Memory Complexity | O(1) |

---

## 8. Step 2 – Medical Token Reconstructor

**Purpose**

Reconstruct fragmented medical terminology generated by ASR systems.

**Examples**

```
met + for + min  →  metformin
hyper + tension  →  hypertension
```

**Responsibilities**

- Detect fragmented words
- Search neighboring tokens
- Validate timestamps
- Merge fragments
- Update confidence values
- Preserve transcript metadata

**Workflow**

```
Read Token
  ↓
Fragment Detection
  ↓
Neighbor Search
  ↓
Timestamp Validation
  ↓
Merge Tokens
  ↓
Return Updated Token
```

**Complexity**

| Metric | Value |
|---|---|
| Time Complexity | O(n) |
| Memory Complexity | O(n) |

---

## 9. Step 3 – Timestamp Matcher

**Purpose**

Matches corresponding transcript tokens generated by different ASR systems.

**Matching Criteria**

- Same speaker
- Similar timestamps
- Similar text (when applicable)

**Workflow**

```
Whisper Token
  ↓
Timestamp Lookup
  ↓
Medical Transcript Search
  ↓
Nearest Match
  ↓
Return Matched Pair
```

**Features**

- Supports configurable timestamp thresholds
- Preserves speaker information
- Handles unmatched tokens gracefully

**Complexity**

| Metric | Value |
|---|---|
| Time Complexity | O(n log n) |

---

## 10. Step 4 – Needleman-Wunsch Aligner

**Purpose**

Performs optimal sequence alignment between transcript outputs.

**Why Alignment?**

Different ASR systems tokenize medical terms differently.

```
hyper tension  →  hypertension
```

**Technologies Used**

- Dynamic Programming
- RapidFuzz lexical similarity
- Timestamp similarity
- Confidence weighting

**Workflow**

```
Matched Tokens
  ↓
Similarity Calculation
  ↓
Dynamic Programming Matrix
  ↓
Traceback
  ↓
Aligned Transcript
```

**Advantages**

- Robust sequence alignment
- Handles insertions and deletions
- Handles spelling variations
- Preserves transcript ordering

**Complexity**

| Metric | Value |
|---|---|
| Time Complexity | O(m × n) (optimized using local timestamp windows) |

---

## 11. Step 5 – MedGemma

**Purpose**

Generate the final corrected transcript and structured clinical summary.

**Responsibilities**

- Read environment variables
- Build prompts
- Create JSON payload
- Send API requests
- Parse responses
- Return corrected transcript and summary

**Environment Variables**

- `MEDGEMMA_API_URL`
- `MEDGEMMA_API_TOKEN`
- `MEDGEMMA_MODEL`

**Workflow**

```
Aligned Transcript
  ↓
Prompt Builder
  ↓
HTTP Request
  ↓
MedGemma
  ↓
Response Parser
  ↓
Corrected Transcript
  ↓
Clinical Summary
```

**Clinical Summary Sections**

- Chief Complaint
- History
- Assessment
- Plan

**Error Handling**

- Invalid API key
- API timeout
- Invalid JSON response
- Empty response
- Retry mechanism

## 16. Exception Handling

Each module handles its own failures independently.

| Failure | Behavior |
|---|---|
| Translation failure | Preserve original text |
| Reconstruction failure | Preserve original token |
| Timestamp matching failure | Mark token as unmatched |
| API failure | Retry request |
| Parsing failure | Log error and return appropriate response |

## 20. Current Limitations

- Requires access to a MedGemma API endpoint
- Depends on internet connectivity for LLM inference
- Long transcripts increase prompt size and response latency
- Rare or unseen medical terminology may require manual verification
- Translation quality depends on language detection accuracy
- Needleman-Wunsch alignment becomes computationally expensive for very large transcript windows

## 23. Conclusion

The Medical Transcript Processing Pipeline is a modular, scalable, and healthcare-focused post-processing framework designed to improve the quality of ASR-generated medical transcripts. By combining language translation, text normalization, medical token reconstruction, timestamp-based matching, Needleman–Wunsch sequence alignment, and MedGemma-based transcript correction and summarization, the pipeline transforms raw speech transcripts into accurate, structured, and clinician-ready documentation while preserving important metadata such as speaker labels and timestamps. It provides a robust architecture suitable for medical documentation workflows and integration with healthcare information systems.
